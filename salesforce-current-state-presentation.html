Summary
Current State at a Glance
06

The following observations reflect the current state of the Salesforce delivery platform across the five reviewed areas.

01 · Code & Repository
  • Self-hosted Bitbucket — all code on internal JPMC AWS infrastructure
  • 3 separate repositories for metadata, Docker image, and shared library
  • No native Salesforce org-diff or org tracking tooling in use
  • Image versioning requires a manual promotion step via Artifactory
02 · Release & Branching
  • Trunk-based model with sprint-boundary release cuts
  • One release branch per sprint — no parallel release streams
  • Release branch creation is a manual step at sprint end
  • A single release spans two sprints from cut to production
03 · Deployment Pipeline
  • Two triggers: PR Open (validation) and PR Merge (deploy)
  • All CI runs inside a Docker container via Jenkins/Cloudbees
  • 5 quality gates enforced automatically before any deployment
  • Pipeline defined in Jenkinsfile using the Dynamo Shared Library
04 · Environments
  • 3 shared sandboxes (INT, E2E, NFT) plus personal dev orgs per developer
  • No environment drift detection between deployments
  • Linear promotion — no parallel release stream support today
  • E2E → NFT promotion is a manual coordination step between sprints
05 · Change Management & Governance
  • SNOW Change Request creation is fully automated from the CI/CD pipeline
  • Test evidence automatically captured in JET for every release
  • Peer review, change approval, and comms remain manual human checkpoints
  • Post-deploy steps and validation are manual tasks defined in Release Notes
Note
This document reflects the current state only. It is intended to inform the leadership team's evaluation of where tooling such as Copado may add value across code management, release automation, environment management, and governance.
