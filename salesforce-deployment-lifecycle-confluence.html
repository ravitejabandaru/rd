üîç Problems with the Current Approach

Beyond individual manual steps, the current pipeline has four systemic problems that affect the team's ability to release safely and quickly.

‚ñ∂ Problem 1 ‚Äî A single release takes two full sprints from cut to production High Impact

E2E testing happens in sprint N. NFT testing and production deployment happen in sprint N+1. This means every release is at least two sprints long ‚Äî roughly 4 weeks. A bug found in NFT forces a fix into the following sprint, extending to 3 sprints. There is no ability to run parallel release streams, so a late release blocks everything behind it.

TodayImpact
Sprint N ‚Äî E2E testingCode frozen 2 weeks before production
Sprint N+1 ‚Äî NFT + ProductionTotal cycle: 4‚Äì6 weeks from dev to prod
No parallel pipelinesOne blocked release blocks all subsequent releases
‚ñ∂ Problem 2 ‚Äî No environment drift detection; failures discovered at deploy time High Impact

There is no automated check to confirm that a target sandbox (E2E, NFT, or Prod) is in the expected state before deploying. If a sandbox has been manually modified, or a previous failed deployment left it in a partial state, this is only discovered when the next deployment fails ‚Äî wasting the deployment window and requiring a rollback.

‚ñ∂ Problem 3 ‚Äî Test evidence is fragmented across JET, JIRA, and manual documents Medium Impact

Apex and Jest results are stored in JET. JIRA holds test case traceability. Selenium and Playwright results live in the CI pipeline. Test exit reports are manually authored Word or Confluence documents. Change approvers must cross-reference all four sources to make an informed approval decision ‚Äî introducing delays and the risk of approving with incomplete evidence.

‚ñ∂ Problem 4 ‚Äî Pipeline toolchain requires a separate Docker + Shared Library + Artifactory maintenance track Medium Impact

The CI/CD capability depends on a Docker image, an Artifactory pipeline, and a shared library that all need to be maintained independently of the Salesforce codebase. Image updates require a manual promotion step and a manual sfdx-app.yaml version bump. This is operational overhead that does not exist in a Salesforce-native DevOps platform.

üéØ Recommendation

Based on the analysis above, the team recommends initiating a Copado proof-of-concept focused on the three highest-impact, lowest-effort improvements. A full platform migration is not proposed at this stage.

üéØ Recommended Approach ‚Äî Phased Adoption

Start with the three quick wins that directly address the highest-risk manual steps on the critical path to production. Evaluate results before committing to a full platform migration.

01
Proof of Concept
Automate environment promotion (E2E ‚Üí NFT ‚Üí Prod) and org drift detection in an isolated PoC environment. Measure impact on release cycle time.
02
Governance Integration
Replace manual peer review and CR approval with Copado Compliance Hub gates. Integrate with existing ServiceNow change management workflow.
03
Full Pipeline Migration
Based on PoC results, evaluate full migration from Jenkins/Docker/Shared Library to Copado DX. Consolidate test evidence with Copado Testing.
üìù
Note on existing toolchain This recommendation does not require an immediate full migration. The existing GitHub, Jenkins, and Docker pipeline can run in parallel with Copado during the PoC phase, reducing risk and allowing a controlled evaluation against real release cycles.
