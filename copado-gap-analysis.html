Section 3 — Gap analysis
Current Problems → What Copado Addresses
✕
Current Problem

What Copado enables
Release Management
Manual release branch creation at every sprint end
A developer must manually cut the release branch from Dev. If missed or delayed, the entire release shifts by a sprint. No tooling enforces this step.
High Risk
Copado Pipelines
Automated promotion pipelines triggered by sprint events
Copado's native pipeline model replaces manual branch cutting. Promotion from one environment to the next is defined as a pipeline step and can be triggered automatically or with a single click.
Copado Pipelines
Environment Promotion
Manual E2E → NFT promotion with no drift detection
Promotion relies on sprint coordination. No tooling checks if the target environment is in the expected state before deploying. Drift is discovered at deployment time.
High Risk
Copado Org Difference
Automated org diff before every promotion
Copado's Org Difference tool compares the source and target org state before promoting. Drift is caught before deployment, not after. Promotions become a governed pipeline step.
Copado Org Diff
Change Management
Manual peer review and change approval — no structured enforcement
Reviewers manually check both the GitHub PR and the ServiceNow CR. There is no structured checklist, no enforcement gate, and approval delays block the entire deployment window.
High Risk
Copado Compliance Hub
Structured approval gates built into the pipeline
Copado Compliance Hub enforces approval gates as pipeline steps. Compliance rules, checklists, and evidence requirements are defined once and applied consistently to every release — no manual coordination needed.
Copado Compliance Hub
Post-Deployment
Manual post-deploy steps from Release Notes — no audit trail
Post-deployment steps are defined in a Release Notes document and executed manually. There is no automated execution, no verification that steps were completed, and no audit trail linked back to the CR.
High Risk
Copado Functions / Pipeline Steps
Post-deployment steps as automated pipeline functions
Copado Functions allow post-deployment steps to be defined as versioned, repeatable pipeline steps. Every execution is logged and linked to the deployment record — providing a full audit trail automatically.
Copado Functions
Source Control Structure
3 separate repos — metadata, Docker image, shared library
Salesforce metadata, Docker image builds, and the shared library are maintained in 3 separate repositories. Changes to any one require manual coordination with the others — especially image promotions via Artifactory.
Medium Risk
Copado DX / Native Git Model
Unified Salesforce-native Git model — no separate toolchain repos
Copado DX manages Salesforce metadata, versioning, and deployments natively — removing the need for a separate Docker container and shared library. The pipeline toolchain is embedded in Copado itself.
Copado DX
Testing & Evidence
Test evidence spread across JET, JIRA, and manual reports
Test results live in JET, traceability in JIRA, exit reports as manual documents, and selenium/playwright results in the CI pipeline. No single source of truth for a release's test evidence.
Medium Risk
Copado Testing
Unified test management and evidence linked to every deployment
Copado Testing consolidates test execution, results, and traceability in one place. Every test run is linked to a specific pipeline run and deployment record — producing a complete, auditable evidence trail automatically.
Copado Testing
Release Cadence
A single release takes two full sprints from cut to production
E2E testing in sprint N, NFT testing and production deployment in sprint N+1. No parallel release streams. A bug found in NFT can delay the entire next release cycle.
High Risk
Copado Pipelines
Parallel pipelines and faster promotion cycles
Copado supports multiple parallel pipelines, enabling concurrent releases. Automated promotions and quality gates reduce the time from code merge to production — without sacrificing governance.
Copado Pipelines
Deployment Comms
Manual notifications to Contact Centre — timing errors possible
The comms team manually drafts and sends deployment notifications. There is no system-driven trigger — meaning comms can be sent too early, too late, or missed entirely.
Medium Risk
Copado Notifications / Functions
Event-driven notifications triggered by pipeline state changes
Copado can trigger notifications automatically when a deployment reaches a specific pipeline stage — ensuring comms are sent at exactly the right time, consistently, with no manual intervention required.
Copado Notifications
